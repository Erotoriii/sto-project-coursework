# ADR-002: Вибір монолітної архітектури

## Статус
Accepted

## Контекст
Команда розробки: 1-2 студенти/розробники.
Термін реалізації: 3-4 місяці.
Складність бізнес-логіки: середня (каталог фільмів, схема залу, кошик, оплата).
Очікуваний час відповіді інтерфейсу касира — не більше 200 мс для швидкого обслуговування черги.

## Рішення
Використовувати монолітну архітектуру з чітким поділом на модулі (Presentation → Service → Data Access) та виділеним REST API.

## Обґрунтування
1. Швидкість розробки: немає потреби витрачати час на налаштування інфраструктури для десятків мікросервісів та їх синхронізацію.
2. Простота розгортання: вся система (бекенд кінотеатру) розгортається на одному сервері єдиним пакетом.
3. Простота транзакцій: ACID-транзакції при продажу квитків набагато легше забезпечити в межах одного додатка та однієї БД.

## Альтернативи
| Варіант | Чому відхилено |
|---------|---------------|
| Мікросервіси | Занадто складно для поточного масштабу; потрібна окрема команда DevOps; overhead на мережеві виклики |
| Serverless (AWS Lambda) | Vendor lock-in; холодний старт критичний для інтернет-магазину; вища вартість при стабільному навантаженні |

## Наслідки
### Позитивні
- Швидкий старт розробки та випуск MVP-версії проєкту.
- Зручне інтеграційне тестування всієї системи цілком. 
### Негативні
- Усі компоненти (афіша, бронювання, звіти) масштабуються тільки разом (монолітно).
